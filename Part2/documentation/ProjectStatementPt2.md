## 1 Introduction

The project is formative and summative, and it is up to the students to ask for feedback. It has two parts:

1. **1st part:** 60% of final TECAA classification. Group work with individual evaluation.
2. **2nd part:** 40% of final TECAA classification. Group work with individual evaluation.

Changes in group composition are allowed and preferentially communicated to the TECAA teacher until April 23. They must be approved.

---

## 2 Description

This chapter establishes the context and motivation for the second phase of the TECAA project.

### 2.1 Business context

The scenario involves a software development company specialising in static websites. Having successfully integrated the Hugo generator into their workflow, the company is now expanding its services into the e-learning sector.

Following the completion of the documentation websites in the first phase, each team must:

- Prepare, conduct, and evaluate a **Hands-on Workshop (HW)**.
- Update their documentation site to include HW pages (Main page, Prerequisites + presentation, Steps with Twine stories, Questionnaire).

#### 2.1.1 Accessibility and performance

Each member must produce and compare two versions of assigned content to demonstrate skills in improvement.

##### 2.1.1.1 Accessibility

Compliance with WCAG 2.2 must be evaluated. Level A is mandatory; AA is aspirational. Each developer must include an accessibility statement.

##### 2.1.1.2 Performance

Focus on Core Web Vitals (CWV). Use laboratory data if real-user data is unavailable. Describe the lab environment in detail for replication.

#### 2.1.2 Goal Question Metric approach

Measurements must be goal-oriented using the GQM approach.

- **Goal 1:** Evaluate HW effectiveness and participant satisfaction.
- **Goal 2:** Ensure compliance and inclusive UX (accessibility/performance).

| GQM Question (Goal 1)             | Metric           |
| :-------------------------------- | :--------------- |
| Q1.1: Success of technical tasks? | Detailed by team |
| Q1.2: Instructor support quality? | Detailed by team |

| GQM Question (Goal 2)                           | Metric           |
| :---------------------------------------------- | :--------------- |
| Q2.1: Level of accessibility compliance?        | Detailed by team |
| Q2.2: User experience (loading, interactivity)? | Detailed by team |

### 2.2 Academic requirements

#### 2.2.1 Hands-on workshop

**Rule 2.2.1:** Tasks requiring per-person work + group vote must be completed individually first, then voted on by the group.

##### 2.2.1.1 HW motivational material

Develop an initial image via generative AI, then process it into a final image. Insert into mockups (mug, poster, etc.). Document the prompts and process in the individual report.

##### 2.2.1.2 Questionnaire for HW

Collect user opinions via questionnaire. Task completion must be verified through other modalities (logs, completion rates).

##### 2.2.1.3 AI generated presentation document

Prepare an English presentation using two AI tools for comparison. Document methodology, prompts (which must differ between members), and apply Mayer’s principles of Multimedia Learning.

##### 2.2.1.4 Other issues

Workshop duration: 50-60 minutes. Date: **May 12th**. Times: 6:10 PM (T100), 7:10 PM (T101), 8:50 PM (T102).

#### 2.2.2 Artefacts in the repository

Consistency across artefacts is valued. Adopt best practices for commit messages. All work must be auditable via the Git repository.

##### 2.2.2.1 Individual and group reports

Reports must be in Markdown or LaTeX to allow version tracking. **.docx files are not allowed**. History of changes from each student must be visible in the Git repository.

## 3 Suggested process

- **Activity 1:** Work division, GQM plan, AI image generation, AI presentation generation.
- **Activity 2:** Website updates, accessibility/performance checks, questionnaire development.
- **Activity 3:** Realise workshop, gather data (logs, traces, completion rates), analyze results.
- **Activity 4:** Finalize GQM analysis, complete reports, global revision.

## 4 Terms and conditions

### 4.1 Feedback and guidance

Regular feedback during lab classes is mandatory. Groups must provide frequent updates.

### 4.2 Submission

Submit a .zip file of the repository to Moodle with signed integrity statements. AI use must be identified in a dedicated "Use of AI-generated content" section.

### 4.3 Evaluation

- **Group component (25%)**
- **Individual component (75%)**
  Commits must reflect individual contributions. All members must be able to answer questions about any part of the project.
